# Pishposh — Project Context for Claude

## Stack

- **Server**: Express, Kysely (query builder), PostgreSQL, TypeScript. Runs via `tsx`. Auth is JWT Bearer tokens in the `Authorization` header.
- **Client**: React + Vite + TypeScript.

## Conventions

- No semicolons anywhere in TS/TSX.
- Singular folder names for domain modules: `src/models/post/post.ts`, not `src/models/posts/`.
- DB uses `snake_case`; app layer uses `camelCase`. Conversion happens in `dbRowTo*` functions at the model boundary.
- IDs are generated by Postgres via `gen_random_uuid()` defaults — do not generate UUIDs in app code.
- Generated columns and triggers preferred over app-side computation when it keeps data consistent.

## Architecture

- **Route layer** handles HTTP concerns: auth middleware, input validation (UUID shape, body schema), existence checks, status codes.
- **Model layer** assumes trusted inputs and returns domain results. No HTTP awareness.
- Vote counts (`upvotes`, `downvotes`) on `posts` are maintained by a trigger on `post_votes`. `score` is a generated column (`upvotes - downvotes`).
- `viewerVote` is per-request context, not stored — fetched via correlated subquery in feed/read queries.

## Review Focus

Please focus on substantive issues; skip style nits the linter would catch.

- Correctness: missing `return` after Express response calls is a recurring footgun worth flagging.
- Security: confirm all mutation routes require auth; read routes should use `req.token?.userId ?? null` for optional viewer context.
- Type safety: flag any `as any`, non-null assertions without justification, or places Kysely types are bypassed.
- Consistency with the conventions above.
- Performance issues that would bite at real scale — but flag them as "worth knowing, not urgent" unless they're actually hot paths.
